# Data Import Strategy

## Overview
This document outlines how we import and organize data into the SQLite knowledge base.

### What we import

| Source | Table | Scope |
|--------|--------|--------|
| **doc.json** | functions | All JSDoc-documented API from `packages/` (generated by `pnpm run jsdoc-json`). Excludes private names (`_*`) and `kind: 'package'`. |
| **website/public/*.json** | presets | Six sample-bank files: tidal-drum-machines, uzu-drumkit, uzu-wavetables, vcsl, mridangam, piano. |
| **Built-in synths** | presets | Names from `registerSynthSounds()` and `registerZZFXSounds()` (waveforms, sbd, supersaw, bytebeat, pulse, noises, ZZFX). |
| **Inline Dirt-Samples** | presets | Bank names from website prebake (casio, crow, insect, wind, jazz, metal, east, space, numbers, num). |
| **Recipes (tunes/examples)** | recipes | From JS/MJS tune files (see below) and from **MDX**: `website/src/pages/recipes/*.mdx` (recipes.mdx, arpeggios.mdx, rhythms.mdx, microrhythms.mdx). Each `<MiniRepl tune={\`...\`} />` snippet is extracted; title uses page title + section (## heading). |

### What we do not import

- **Functions**: Exports that donâ€™t appear in doc.json (e.g. some codemirror internals, or files that fail JSDoc parse). See `undocumented.json` for a report.
- **Presets**: Soundfont presets (from `registerSoundfonts()`), user samples from IndexedDB, or samples loaded from external URLs (e.g. `samples('github:...')`) that arenâ€™t in the six JSON files above.
- **Recipes**: Only the listed JS/MJS tune files and the MDX files under `website/src/pages/recipes/` are imported; other MDX (e.g. workshop, learn) are not.

## Categorization Strategy

### Function Categories (based on filename and usage)

1. **pattern** - Pattern constructors
   - Files: `pattern.mjs` (constructors: cat, seq, stack, stepcat, etc.)
   - Examples: `cat`, `seq`, `stack`, `stepcat`, `arrange`, `silence`

2. **time** - Time modifiers
   - Files: `pattern.mjs` (time functions)
   - Examples: `slow`, `fast`, `early`, `late`, `rev`, `iter`, `euclid`

3. **control** - Parameter/control functions
   - Files: `controls.mjs` (most functions)
   - Examples: `s`, `note`, `n`, `gain`, `cutoff`, `pan`

4. **effect** - Audio effects
   - Files: `dough.mjs`, `superdough.mjs`
   - Examples: `lpf`, `hpf`, `distort`, `reverb`, `delay`

5. **signal** - Signal generators
   - Files: `signal.mjs`
   - Examples: `sine`, `saw`, `square`, `noise`

6. **utility** - Utility functions
   - Files: `util.mjs`, `repl.mjs`, `pick.mjs`
   - Examples: `setcpm`, `pick`, `getFreq`

7. **motion** - Motion/device input
   - Files: `motion.mjs`
   - Examples: `accelerationX`, `gravityX`

8. **other** - Everything else
   - Files: `euclid.mjs`, `voicings.mjs`, etc.

## Data Mapping

### doc.json â†’ functions table

| doc.json field | database field | Transformation |

### Category Mapping Logic

```python
def get_category(meta_filename):
    mapping = {
        'pattern.mjs': 'pattern',  # Will need to distinguish constructors vs time modifiers
        'controls.mjs': 'control',
        'signal.mjs': 'signal',
        'dough.mjs': 'effect',
        'superdough.mjs': 'effect',
        'motion.mjs': 'motion',
        'util.mjs': 'utility',
        'repl.mjs': 'utility',
        'pick.mjs': 'utility',
        'euclid.mjs': 'time',  # Time-related
        'voicings.mjs': 'control',  # Note-related
    }
    return mapping.get(meta_filename, 'other')
```

### Special Cases

1. **Pattern.mjs functions**: Need to distinguish:
   - Pattern constructors (cat, seq, stack) â†’ category: 'pattern'
   - Time modifiers (slow, fast, rev) â†’ category: 'time'
   - Solution: Check function name against known lists

2. **HTML in descriptions**: Strip HTML but preserve structure
   - Use `html.parser` or regex to clean
   - Keep important info (code examples, links)

3. **Missing fields**: Handle gracefully
   - `params`: Use empty JSON array `[]`
   - `examples`: Use empty JSON array `[]`
   - `synonyms`: Use empty JSON array `[]`

## Data Cleaning Rules

1. **Filter out**:
   - Functions starting with `_` (private)
   - Functions without `name`
   - Functions with `kind: 'package'`

2. **Normalize**:
   - Strip HTML from descriptions
   - Ensure JSON arrays are valid JSON
   - Trim whitespace from strings

3. **Validate**:
   - Check required fields exist
   - Verify JSON is valid before storing
   - Ensure unique function names

### Sample-Bank JSON Structure

- **Flat**: Top-level keys (except `_base`) are preset names; value is an array of sample paths.
- **Nested**: If a value is a dict (e.g. piano bank with note â†’ file), the key is treated as one preset name.
- `_base` is skipped (URL prefix for samples).

### presets Table Mapping

| Source | database field | Notes |
|--------|----------------|--------|
| key name | `name` | Preset identifier, used in `s("name")` |
| (fixed) | `type` | `"sound"` |
| source file + category | `description` | Short description |
| generated | `code_example` | `s("name")` |
| per-file | `category` | drum-machines, percussion, wavetables, instruments |
| `[source_filename, category]` | `tags` | JSON array |

### Preset Import Process

1. For each `(path, category)` in `PRESET_SOURCES`, load the JSON.
2. Skip `_base`. For each other key: if value is list or dict, add preset with that key as name.
3. Upsert by `name` (update existing or insert).
4. Set `type`, `category`, `description`, `code_example`, `tags`.

## Recipe Import (Tunes and Examples)

### Recipe sources

| Path | Content |
|------|--------|
| `website/src/examples.mjs` | Array of 3 full tunes (export const examples = [ \`...\`, ... ]). |
| `website/src/repl/tunes.mjs` | Named exports (export const name = \`...\`) â€” dozens of tunes. |
| `examples/codemirror-repl/tunes.mjs` | Same pattern as repl/tunes.mjs. |
| `examples/minimal-repl/tune.mjs` | Single default export (\`...\`). |
| **MDX** `website/src/pages/recipes/*.mdx` | Each \`<MiniRepl tune={\`...\`} />\` is one recipe. Title = frontmatter \`title\` + \` â€“ \` + last \`##\` section; category = \`recipe\`. |

### Extraction rules

- **Title**: From the first comment line (e.g. `// "Title" @by author` or `// Title`) or from the export name (e.g. `giantSteps` â†’ "giant steps").
- **Code**: Full template literal body (leading/trailing whitespace trimmed).
- **Category**: `tune` for all.
- **Tags**: JSON array including `import:<source>` (e.g. `import:website-tunes`) so re-import replaces only these rows.

On each run, recipes whose `tags` contain `import:` are deleted, then all current recipe sources are re-parsed and re-inserted. User-created recipes (no `import:` tag) are left unchanged.

## Import Process

1. **Functions**: Load doc.json â†’ filter â†’ categorize â†’ clean â†’ insert/update functions table.
2. **Presets**: Load each sample-bank JSON and built-in lists â†’ upsert presets table.
3. **Recipes**: Delete previously imported recipes (tagged with `import:`), then load each recipe source file â†’ extract tunes/examples â†’ insert recipes table.
4. **Report statistics** (imported/skipped/errors for each).

## Validation Checklist

After import, verify:

**Functions**

- [ ] All functions from doc.json are imported (except filtered ones)
- [ ] Categories are correctly assigned
- [ ] JSON fields are valid JSON
- [ ] No duplicate function names
- [ ] Examples, synonyms, and parameters are preserved

**Presets**

- [ ] All preset names from the sample-bank JSONs and built-in lists are present in the database
- [ ] No duplicate preset names
- [ ] Preset categories match sources (drum-machines, percussion, wavetables, instruments, synth, samples)
- [ ] Preset `tags` field is valid JSON

**Recipes**

- [ ] All recipes from the four recipe source files are present (website examples, website tunes, codemirror tunes, minimal tune)
- [ ] Recipe `tags` contain the expected `import:*` value
- [ ] Recipe `code` is non-empty and `title` is set

Run `python validate_import.py` from the `backend/` directory to execute the full checklist.
